# ManualEngine Systems And Contracts

This guide is the quick reference for what the prototype currently supports and how the major systems are expected to be wired together. Consult it before adding features that cross subsystem boundaries, and update it whenever a new feature adds an interface contract, ownership rule, or initialization dependency. Future message-driven systems should also follow `docs/system_inboxes.md`.

## Available Gameplay Systems

- **World ownership:** `Engine::World` owns transient world object handles, transforms, stable object IDs, local/world CPU bounds, collision flags, angular velocity, and optional renderer instance bindings.
- **Stable identity:** `Engine::ObjectId` is the persistence-facing identity. `WorldObjectHandle` is transient and must not be saved.
- **Scene kernel:** `Engine::Scene` owns renderer-independent scene actor records, metadata-only component records, scene-local transform hierarchy state, and a CPU-only lifecycle/tick scheduler shell. `SceneActorHandle`, `SceneComponentHandle`, and `SceneSystemHandle` are transient generation-counted runtime handles; `SceneObjectId` is the stable scene identity used by the Phase 13 core scene serialization path. Scene transform hierarchy and scheduler state remain scene-runtime state and do not drive `World`, physics, or navigation implicitly. Scene runtime adapters should register scheduler systems and tick through `Scene::tickFrame()`/`tickFixed()` instead of adding hidden App-local update ordering.
- **Scene render bridge:** `Engine::SceneRenderBridge` owns scene mesh, skinned mesh, light, and camera component records plus mappings to live renderer handles through an explicit `SceneRenderBackend`. The bridge syncs scene world transforms to renderer-facing resources during `PreRender` or manual sync; Renderer remains handle-based and does not read scene actor/component storage.
- **Scene authored adapter:** `Engine::adaptImportedSceneToScene` consumes already-imported `Assets::Assimp::ImportedScene` CPU data and creates scene actors, hierarchy links, adapter-owned static mesh/material/texture resources, and scene render bridge mesh/light components. The release authored sample path uses this adapter for static authored scenes; debug authored mode keeps the existing `PartitionedAuthoredScene`, authored cache, and async loading path for diagnostics and streaming validation.
- **Scene animated adapter:** `Engine::SceneAnimatedModelAdapter` consumes already-imported animated `Assets::Assimp::ImportedScene` CPU data, creates scene actors for imported nodes, skeleton/joint bindings, animator records, adapter-owned skinned mesh/material/texture resources, and scene render bridge skinned mesh components. It reuses the shared `AnimatedModel` pose sampling, playback, palette, and skinned vertex packing helpers; the release authored sample uses this adapter for its KayKit animated character while existing debug `AnimatedModel`, cache, async, renderer skinned mesh, and App animated sample paths remain operational.
- **Scene physics:** `Engine::ScenePhysicsWorld` is a Jolt-backed, scene-owned physics facade composed beside `Engine::Scene`. It owns generation-counted body/collider handles, explicit shape descriptors, fixed-phase transform sync, raycast/sweep/overlap/closest-point query APIs, diagnostics, and renderer-independent debug request records. Jolt types stay private to `src/Engine/Physics`; physics handles are distinct from scene actor/component, renderer, world, and navigation handles.
- **Scene character movement:** `Engine::SceneCharacterMovementSystem` is a headless scene gameplay system composed beside `ScenePhysicsWorld` and optional `SceneNavigationService`. It owns transient character handles and one kinematic capsule physics body/collider per character, updates only during fixed movement calls or its registered fixed scene system, and never creates terrain colliders, nav tiles, renderer resources, App commands, serialization files, or chunk streaming work implicitly.
- **Scene/world migration bridge:** `Engine::SceneWorldMigrationBridge` is a transitional runtime-only bridge for the procedural sample. It maps legacy `WorldObjectHandle` records to scene actors, can bind mapped actors to scene characters or static scene-physics box colliders, and can sync transforms in either direction so existing renderer instances, picking, selection, and save UI keep using `World` while scene systems are evaluated. Bridge mappings, scene handles, physics handles, character handles, and collider bindings are transient and must not be serialized.
- **Modern default scene runtime:** The normal Debug and Release App startup path is scene-backed. It composes `Scene`, scene scheduler systems, scene render bridge, scene authored/animated adapters, `TerrainDataset`, terrain render/navigation/physics adapters, `ScenePhysicsWorld`, `SceneNavigationService`, and `SceneCharacterMovementSystem`. Legacy procedural `World`/`ActorController`/`BlockingCollisionSystem`, legacy `TerrainSystem`, legacy authored streaming, and legacy `AnimatedModel` startup paths are compatibility modes selected explicitly from the command line, not defaults.
- **Open-world streaming contract:** `Engine::OpenWorldStreaming` defines the residency states, dirty flags, transition lanes, payload kinds, durable chunk manifest records, optional variant IDs, per-payload and per-profile halo policies, planning diagnostics, async cache-halo read/decode coordination, optional derived-generation coordination, budgeted live-halo promotion/demotion coordination, and plain queue/cache/resource counters for future modern streaming. Halo planning classifies durable terrain chunks, future scene chunks, and asset dependencies as desired residency only; variant IDs let one durable chunk carry separate records for low/high LODs, far metadata, texture-mip variants, and future behavior/animation/audio records. Predictive focus may prefetch cache records from velocity or goal/path hints, but it does not request live promotion unless a policy explicitly allows it. The cache halo may enqueue `DiskReadDecode` jobs and store CPU cache hits for terrain chunk payloads, terrain render LOD payloads, navigation tile bytes, terrain physics collider payloads, scene chunk records, and metadata-only assets, but it must not upload renderer resources, insert nav tiles, create physics bodies/colliders, activate scene actors, mutate App gameplay, or serialize runtime state. The derived-generation halo may enqueue worker generation jobs only when policy explicitly enables generation-on-miss or refresh; those jobs write/read derived cache payloads and return plain CPU payloads, never live resources. The live halo may enqueue `MainThreadPromote`/`MainThreadDemote` work through `FrameBudget`/`MainThreadWorkQueue`, but actual renderer, navigation, physics, scene, or asset mutation is callback-owned and main-thread-only. Streaming records may store runtime promotion tokens for demotion callbacks, but must never store live subsystem handles or durable serialization payloads.
- **Open-world asset dependency streaming:** `Engine::OpenWorldStreamingAssets` is the narrow adapter from durable asset dependency records to live `AssetCache` acquisitions. Asset dependencies are keyed by `AssetId` plus import settings; cache-halo warming stores metadata payloads only. Live promotion may acquire `StaticMesh` and `Texture` dependencies through `AssetCache` on the main thread and demotion releases those acquisitions exactly once after the last live reference. Materials, skinned meshes, authored scenes, animation clips, and terrain material sets can be represented as metadata dependencies but are not live promotion targets until their ownership becomes streamable. Runtime `AssetHandle`, `CachedStaticMesh`, `CachedTexture`, renderer handles, and streaming runtime tokens must not be serialized.
- **Open-world scene chunk streaming:** `Engine::OpenWorldStreamingSceneChunks` is the narrow adapter from durable scene chunk records to live `Scene` actor/component records. Scene chunk cache-halo reads use the Phase 13 scene binary format and produce CPU `StreamingSceneChunkPayload` records only. Active-halo promotion is main-thread callback-owned: it validates cached scene data, recreates core actors with stable `SceneObjectId`, restores local transforms/hierarchy, and attaches metadata-only components. Demotion destroys only chunk-owned actors created by that binding. `Global` and `Migratory` ownership policies prevent duplicate stable actors across overlapping chunks, but automatic position-based migration remains deferred. Scene chunk manifests and payloads must not contain `SceneActorHandle`, `SceneComponentHandle`, `OpaqueHandle`, renderer, physics, nav, terrain, or asset-cache runtime handles.
- **Open-world streaming runtime:** `Engine::OpenWorldStreamingRuntime` is the S8 coordinator that makes the streaming contracts usable by the modern default scene. It validates a saved streaming build manifest, triggers an explicit heightmap-derived rebuild when the saved build is missing/stale/version-mismatched, runs halo planning, queues cache reads/generation, and schedules callback-owned main-thread live promotion/demotion. The default scene owns the concrete callbacks for renderer terrain handles, navigation tiles, terrain physics colliders, scene chunks, and supported asset dependencies. Saved streaming build manifests and runtime diagnostics remain durable-ID based and must not contain live handles. Heightmap-derived nav tile identity includes the terrain navigation adapter generator version and border-apron settings; changing the border generator, padding, or sample count invalidates old seam-gap tile caches.
- **Reflection and opaque access:** `Engine::ReflectionRegistry` owns typed metadata descriptors for approved runtime properties, and `Engine::SceneReflection` converts transient subsystem handles into `OpaqueHandle` access tokens for tools, native hooks, future serializers, and future Lua bindings. Reflected writes must route through public subsystem APIs and may not return raw storage pointers, renderer resources, Jolt IDs, Detour types, bgfx state, or App objects. `OpaqueHandle` is runtime-only and must not be serialized; durable references remain `SceneObjectId`, `AssetId`, and terrain source/chunk stable IDs.
- **Scene serialization:** `Engine::SceneSerialization` owns the CPU-only binary scene container contract. The current format uses magic `MESCENE\0`, little-endian fixed-width fields, a fixed header, a chunk directory, 64-bit FNV-1a checksums, header-only inspection, whole-file read/write, validation, and restore for core `Scene` actor records, local transforms, hierarchy links, and metadata-only scene components. It may record reflection schema metadata, `AssetId` references, and `TerrainSerializationPrep` durable terrain references, but it must not serialize `OpaqueHandle`, transient runtime handles, renderer resources, Jolt/Detour/bgfx state, App state, or live terrain/nav/physics resources. Apply must validate the snapshot before mutating a live scene. Detailed binary, identity, determinism, and extension rules live in `docs/scene_runtime/serialization_conventions.md`.
- **Scene native behavior hooks:** `Engine::SceneBehaviorHooks` owns CPU-only native C++ behavior callback registrations layered over `Scene`, the scene scheduler, reflection, and opaque handles. Behaviors bind explicitly to scene, actor, component, system, or custom opaque targets and execute through scheduler lifecycle/tick ordering; they mutate state only through public APIs or reflected setters. `SceneBehaviorHandle`, `OpaqueHandle`, callbacks, lambdas, and captured runtime objects are transient and must not be serialized. Phase 14 does not add Lua, hot reload, plugin loading, editor UI, App gameplay migration, or automatic procedural `World` hooks.
- **Scene Lua scripting:** `Engine::SceneLuaRuntime` owns one sandboxed Lua VM composed beside `SceneBehaviorHooks`, `SceneReflection`, and `OpaqueHandle`. Scripts compile into per-script environments, register Lua-backed native behavior hooks, execute synchronously through scene scheduler ordering, and can read/write only approved reflected properties through the allowlisted `engine` table. The Lua runtime opens deterministic base/table/string/math libraries only; it does not expose `os`, `io`, `debug`, package loading, renderer/bgfx, Jolt, Detour, App objects, filesystem writes, or subsystem storage. `LuaScriptInstanceHandle`, Lua registry refs, closures, script tables, VM state, and `OpaqueHandle` values are runtime-only and must not be serialized.
- **Debug visualization:** `Engine::DebugVisualizationCollector` owns renderer-independent, frame-transient debug primitive/report requests with category toggles, per-category/global budgets, distance clipping, severity labels, diagnostics, and plain expanded debug line output. It is a diagnostic request layer only: it must not mutate simulation state, own renderer resources, expose subsystem storage, or serialize debug records. App/Renderer composition remains responsible for choosing enabled categories, draining snapshots, converting plain line output to renderer primitives, and showing report rows.
- **Fixed-step loop:** `Engine::FixedStepLoop` provides clamped frame timing and 60 Hz fixed update ticks.
- **Input mapping:** SDL events flow into `Engine::InputState`; `Engine::InputMapping` publishes semantic input actions from `assets/config/input.yaml`.
- **Event queue:** `Engine::EventQueue` carries frame input action events and interaction events. It is explicitly cleared at the end of the frame. It is not a general event bus registry; new message-driven systems should prefer system-owned typed inboxes with publish-only sinks.
- **Camera:** `Engine::OrbitCameraController` supports sim free camera and player-follow mode, with configurable pivot bounds, zoom, pitch, panning, and follow offset.
- **Actors:** `Engine::ActorController` owns kinematic actor state, manual movement, path-following state, facing, terrain grounding, and collision-aware movement.
- **Actor selection and commands:** `Engine::ActorSelection` stores selected actors; `Engine::formationOffsetForActorIndex` provides deterministic simple formation slots.
- **Blocking collision:** `Engine::BlockingCollisionSystem` resolves actor cylinder-vs-world-AABB blocking through the sparse spatial registry. It is a legacy procedural-world kinematic helper, not a physics engine, and should not be wired into future scene-owned physics.
- **Spatial registry:** `Engine::SpatialRegistry` is a sparse XZ grid for debug/gameplay lookup. It does not own world object lifetime.
- **Chunk streaming:** `Engine::ChunkStreamer` owns loaded chunk membership and chunk content handles. App now stages desired chunk loads/unloads around a center point, runs expensive generated-data work through `Engine::AsyncWorkQueue`, and commits live chunks on the main thread with a per-frame budget. Runtime unloads may detach chunk content from loaded membership before phased destruction completes; forced reload/shutdown paths must finish those detached unloads before clearing queued work.
- **Async work queue:** `Engine::AsyncWorkQueue` runs CPU-only jobs that return plain generated data or Detour tile bytes. Workers must not touch live `World`, `Renderer`/bgfx, `TerrainSystem` storage, `NavigationSystem`, `SpatialRegistry`, or mutable save state.
- **Main-thread frame budget:** `Engine::FrameBudget` and `Engine::MainThreadWorkQueue` provide a fixed millisecond budget for deferrable main-thread mutations and resource insertion. Renderer/bgfx calls, live world mutation, live terrain storage, live navmesh insertion, and spatial registry mutation still run on the main thread, but chunk commits and derived rebuilds should be split into small budgeted work items when they can be deferred.
- **Terrain:** `Engine::TerrainSystem` owns the legacy loaded CPU heightfield tiles, biome-shaped generation, height queries, terrain raycasts, terrain diagnostics, optional renderer terrain handles, terrain LOD rebuilds, and navigation build data extraction. `Engine::TerrainDataset` is the renderer-independent terrain runtime owner for source records and loaded CPU chunk records; it supports imported heightmap chunks and procedural chunks through one query path, returns transient generation-counted source/chunk handles, and is the terrain source for the modern default scene. `Engine::TerrainDerivedCache` stores disposable generated CPU payloads for imported terrain chunk heights and renderer-independent terrain LOD meshes under deterministic source/import/settings identity; it uses YAML manifests plus binary payload files, provides worker-safe synchronous helpers and async queue wrappers, and never stores live renderer, navigation, physics, App streaming, or serialization state. `Engine::TerrainRenderLodAdapter` is the renderer LOD build boundary for immutable terrain chunk snapshots; it can consume `TerrainDataset` chunks or legacy `TerrainSystem` tile inputs, probe T3 cache payloads read-only, and generate renderer mesh data on miss/stale/corrupt. Modern App startup commits that output directly to `Renderer::createTerrainTile`; legacy `TerrainSystem` commits remain compatibility behavior. `Engine::TerrainNavigationAdapter` is the navigation build-data boundary for immutable terrain chunk snapshots; it can consume `TerrainDataset` chunks, generated tile data, or legacy `TerrainSystem` tile inputs and produces existing `NavigationTerrainBuildData` only. `Engine::TerrainPhysicsColliderAdapter` is the explicit terrain-to-scene-physics boundary; it builds CPU static triangle mesh collider payloads from authoritative terrain heights, owns transient terrain collider bindings, and creates/releases `ScenePhysicsWorld` static bodies only when callers ask for that handoff. `Engine::TerrainMaterialMetadata` owns in-memory terrain material set, layer, rule, texture dependency, validation, and CPU coverage evaluation contracts. `Engine::TerrainSampleMaterials` provides sample metadata presets for App composition without giving App ownership of terrain material rule construction. `Engine::TerrainMaterialRenderAdapter` is the explicit bridge from that CPU metadata to live renderer terrain material-set resources through `AssetCache`; renderer resources stay live/transient and are not durable material metadata. `Engine::TerrainSerializationPrep` defines the future chunked serialization identity and payload-boundary metadata so source snapshots, edited overrides, and derived cache references can be distinguished without serializing runtime handles. `Assets::loadHeightmapImage` and `Engine::importHeightmapTerrain` add the CPU-only first step for imported heightmap terrain: they decode source height samples and produce terrain chunk descriptors without allocating renderer resources, building navigation data, creating physics colliders, mutating live terrain runtime state, or using `Renderer::loadTexture`. Runtime terrain renderer LOD changes should use worker mesh generation plus budgeted main-thread upload commits so camera movement cannot recreate every visible terrain buffer in one frame.
- **Biomes:** `Engine::BiomeSystem` deterministically samples biome IDs and debug noise values from world coordinates and chunk coordinates.
- **Procedural content:** `Engine::ProceduralChunkContentConfig` produces deterministic prop spawn descriptors from chunk coord, biome rules, archetype IDs, and stable local slots.
- **Archetypes:** `Engine::ObjectArchetypeCatalog` loads YAML object descriptors with visual defaults, bounds, tags, terrain offset, resource metadata, and collision/interaction tags.
- **Persistence:** `Engine::WorldObjectOverrides` stores removed procedural IDs and persistent object overrides; `Engine::WorldState` saves high-level world state only.
- **Persistent editing:** `Engine::PersistentObjectEditor` mutates save-backed overrides for selected object transforms, removal/reset, and placement.
- **Picking and selection:** `Engine::Picking` builds cursor rays and performs CPU object/terrain picking for debug selection and interactions.
- **Interactions:** `Engine::InteractionSystem` turns semantic input plus picking into target-aware interaction events; `Engine::InteractionHandlerSystem` resolves tag-based outcomes.
- **Navigation profiles:** `assets/config/navigation_profiles.yaml` selects one active player-sized agent/build profile for nav tile generation and queries.
- **Navigation local layer:** `Engine::NavigationSystem` wraps Recast/Detour. Runtime tile sync must prefer cache hits, enqueue worker Recast byte builds on misses, and insert only completed tile bytes into the live navmesh on the main thread. Normal gameplay code must not call live `buildTerrainTile()` from the frame path.
- **Navigation runtime API:** `Engine::SceneNavigationService` is a scene-friendly facade over existing live navigation systems. It exposes projection, local path, loaded-tile portal path, reachability, conservative raycast, scene-actor path, diagnostics, and debug request APIs without owning navmesh tiles or triggering tile builds. Recast/Detour types remain hidden behind `NavigationSystem`.
- **Navigation scene geometry:** `Engine::SceneNavigationGeometryRegistry` owns CPU-only scene navigation source records and can snapshot enabled scene static geometry into `NavigationTerrainBuildData` for the existing navigation tile build path. It reads scene actor transforms through public `Scene` APIs, filters scene triangles per requested tile by padded world bounds and configurable world-space walkable slope, keeps terrain base geometry untouched, keeps dirty chunk reporting caller-owned, and never inspects renderer GPU resources or triggers tile builds by itself.
- **Navigation connectivity:** `Engine::NavigationConnectivitySystem` derives loaded-chunk edge portals and neighbor links from local nav tiles and records per-edge portal sampling diagnostics. The modern path samples `NavigationSystem` tile bounds and tile-local nearest-poly queries directly; legacy `TerrainSystem` overloads remain compatibility wrappers. Normal nav tile insert/destroy events should rebuild only changed chunks and neighbors through phased main-thread steps that process chunk setup, one edge or capped sample batch, relinking, and finalization. Full rebuilds are reserved for debug/settings changes but use the same phased mechanism internally. Portal links validate XZ seam distance and max height delta so streamed adjacent tiles can fail gracefully without exposing Recast/Detour internals.
- **World navigation graph:** `Engine::WorldNavigationGraph` builds a deterministic coarse chunk graph over the generated world and returns coarse routes used by actor hierarchical move commands.
- **Navigation cache:** `Engine::NavigationCache` stores derived baseline nav tile bytes, connectivity metadata, and graph metadata under a versioned manifest. Runtime cache file reads/writes use worker-safe helper jobs; cache misses enqueue worker generation rather than forcing synchronous Recast work. Modern scene tile manifests include terrain identity plus scene-geometry source hash, slope threshold, tile padding, and generator version so filtered static-geometry changes invalidate stale Detour bytes.
- **Frame dirty orchestration:** `src/App` owns dirty flags for nav tile sync, nav connectivity, world graph rebuilds, renderer visibility metadata, and picking. Expensive derived systems should run when inputs change, not every frame. Renderer metadata reapply uses dirty chunk sets for local changes and a capped full-reapply queue only when global visibility/debug settings change.
- **Asset registry:** `Engine::AssetRegistry` owns CPU-only asset identity, source metadata, import settings identity, dependency edges, and stale/missing diagnostics. `AssetHandle` is a transient generation-counted runtime handle; `AssetId` is the stable registry identity intended for future scene/component serialization references. Heightmap, terrain source, terrain material set, and terrain chunk records are metadata identities only; generated terrain outputs remain derived data, not live renderer/navigation/physics handles. The registry does not import payloads, allocate renderer resources, replace path-based loaders, or own `AssetCache` entries.
- **Asset cache:** `Engine::AssetCache` deduplicates reusable renderer meshes and textures; terrain draw buffers remain transient.

## Available Rendering Features

- **Renderer resources:** Static meshes, generated fallback cube mesh, runtime textures, solid textures, terrain tiles, materials, fixed-size terrain material sets, render groups, and mesh instances are exposed through renderer-owned handles.
- **Materials:** `Renderer::MaterialHandle` supports base color texture/factor, optional normal texture, metallic, and roughness factors. Material assignment is preferred over texture-only overrides.
- **Scene interface:** `Renderer::RenderView` carries view ID, matrices, camera position, viewport, layer mask, and distance-culling toggle into scene submission.
- **Culling and layers:** Renderer submission supports visibility flags, render layers, frustum culling, max draw distance, and stats for culled/submitted mesh instances and terrain tiles.
- **Render groups:** Renderer render groups tag terrain/instances with chunk metadata for stats and debug grouping only. They never own chunk lifetime.
- **Batching:** Mesh draw items are sorted/grouped by mesh/submesh/material/render state as a conservative CPU batching path.
- **Terrain rendering:** Terrain tiles use generated heightfield meshes with LOD rebuilds and skirts. A tile keeps one material handle as fallback and may also reference a renderer-owned terrain material set with up to four PBR layers and eight rules evaluated in the terrain shader from world height, visual slope, and world X/Z. App procedural terrain currently assigns one shared layered material set while preserving the biome material fallback.
- **Atmosphere:** Renderer-owned sky clear color, exponential distance fog, directional sun color, direction, and intensity.
- **Debug UI:** Dear ImGui presents modern runtime diagnostics and controls for performance, renderer state, atmosphere, camera, scene scheduler, scene render bridge, `TerrainDataset`, modern navigation, scene physics, scene character movement, and debug visualization. Legacy runtime-specific controls for `World`, `TerrainSystem`, `ChunkStreamer`, `ActorController`, `BlockingCollisionSystem`, biome editing, picking, group commands, world graph routing, and legacy save/edit flows are not part of the Debug UI surface; explicit legacy compatibility modes may show only shared renderer/performance/status diagnostics.
- **Debug draw:** Transient colored line/AABB/XZ-rect/frustum primitives visualize modern scene, terrain, navigation, physics, character, and camera diagnostics. Renderer remains the draw consumer; Engine/App composition owns which modern producer requests are submitted.
- **Debug visualization collector:** Engine systems may publish plain debug visualization requests to an explicit collector/sink. Requests are bounded, category-filtered, clipped diagnostics and are not gameplay authority. Renderer debug primitives and ImGui rows are consumers of collector snapshots, not owners of simulation diagnostics.

## Initialization Order

1. **Platform and renderer boot**
   - Create SDL window.
   - Initialize bgfx.
   - Configure renderer vertex layouts.
   - Initialize scene renderer resources.
   - Initialize Dear ImGui debug UI.
   - Read and push initial renderer atmosphere settings.
   - Set initial bgfx reset/view clear/view rect.

2. **Runtime assets and descriptors**
   - `Engine::AssetRegistry` may be created by tools or future scene composition code to track stable source metadata. The current sample boot path still uses path-based imports and `AssetCache` directly.
   - Create `Engine::AssetCache`.
   - Acquire fallback/player/debug solid textures and fallback mesh.
   - Load object archetypes from `assets/config/object_archetypes.yaml`, falling back to defaults on failure.
   - Load biomes from `assets/config/biomes.yaml`, falling back to defaults on failure.
   - Create renderer materials for archetypes and biome terrain colors.
   - Create the optional shared terrain material set for the procedural terrain sample; keep per-tile biome materials as fallback if layered resource creation fails.

3. **Engine services**
   - Load the active navigation profile from `assets/config/navigation_profiles.yaml`, falling back to defaults on failure.
   - Create `World`, `ActorController`, `BlockingCollisionSystem`, `SpatialRegistry`, `TerrainSystem`, `NavigationSystem`, `NavigationConnectivitySystem`, `WorldNavigationGraph`, `NavigationCache`, `WorldObjectOverrides`, `AsyncWorkQueue`, `FrameBudget`, `MainThreadWorkQueue`, `ChunkStreamer`, input/event/interaction systems, and camera. `Scene` is available as an isolated runtime container but is not part of the sample boot path yet.
   - Use one consistent chunk size across spatial registry, terrain, procedural content, chunk streamer, persistent editor, and world graph.
   - Configure camera bounds and terrain LOD distances before creating chunks.

4. **Chunk, nav, and graph bootstrap**
   - Create chunk factory from archetypes, materials, terrain material resolver, overrides, and spatial registry.
   - Queue initial chunks around the camera pivot and commit completed generated chunks under the configured per-frame budget.
   - Insert worker-built or cached loaded nav tile bytes from terrain plus blocking prop bounds as chunks commit.
   - Rebuild loaded navigation connectivity from loaded nav tiles.
   - Rebuild the coarse world navigation graph around the active graph center.
   - Update terrain LODs and apply renderer visibility/material metadata.

5. **Player and demo actors**
   - Create player world object with `ObjectId::player()`, renderer instance, CPU bounds, and actor state.
   - Create demo squad actor objects and register them in world, renderer, actor controller, and spatial registry.
   - Call `World::syncRenderState()` after initial object transforms are valid.

6. **Frame loop**
   - Begin frame timer and input frame.
   - Feed SDL events to Dear ImGui first; only feed game input when ImGui does not capture mouse/keyboard.
   - Update window size/bgfx reset/view settings as needed.
   - Publish semantic input events.
   - Update camera from semantic events and current player target.
   - Stage chunk streaming: enqueue missing desired chunks, cancel stale pending work, enqueue budgeted load/unload commits, and mark derived systems dirty when loaded membership changes.
   - Begin the main-thread frame budget, drain budgeted commit/rebuild work, sync nav tiles, rebuild connectivity/graph, update terrain LODs, and apply renderer visibility only when their dirty flags require it.
   - Run fixed updates: world angular velocity, player actor with real input, demo actors with empty input, spatial registry updates.
   - Sync world transforms to renderer.
   - Update picking and interaction events.
   - Handle movement commands, hierarchical route requests, local path requests, and coarse route debug state.
   - Enqueue debug primitives.
   - Draw scene, draw debug primitives, draw Dear ImGui, submit `bgfx::frame()`.
   - Process debug save/load/rebuild/edit requests and clear the event queue.

7. **Shutdown**
   - Clear navigation/graph/connectivity state.
   - Unload chunks and clear spatial registry.
   - Destroy renderer materials.
   - Release cached textures/meshes and shut down `AssetCache`.
   - Shut down Dear ImGui, scene renderer, bgfx, SDL window, and SDL.

## Cross-System Contracts

- **Scene runtime identity:** Scene actor and component runtime handles are transient, generation-counted values and must never be serialized. `SceneObjectId` is stable scene identity for the core scene binary serialization path and remains distinct from runtime handles. `AssetHandle`, renderer handles, scene render bridge handles, scene physics handles, character handles, navigation handles, terrain runtime handles, and `OpaqueHandle` values are also transient and must not appear in serialized payloads. Existing `ObjectId` remains the procedural world/save identity until an explicit migration is defined.
- **Scene transform hierarchy:** `Scene` actor transforms use local TRS data, cached world matrices, parent/child links, and dirty propagation. Hierarchy mutation is main-thread runtime state only; future authored-scene, renderer, physics, navigation, and serialization phases must consume it through explicit bridge APIs rather than reading storage directly.
- **Scene tick scheduler:** `Scene` scheduler phases are synchronous main-thread callbacks with explicit lifecycle transitions and fixed/variable phase order. Existing `FixedStepLoop` and App frame code still own fixed timestep accumulation and decide when to call scene fixed ticks. Scene runtime adapters should use scheduler systems for normal update flow; the release authored scene sample updates animation in `VariableAnimation` and renderer bridge sync in `PreRender`. Future renderer, physics, navigation, animation, scripting, and behavior systems must register through the scheduler rather than adding hidden app-local ordering.
- **Scene Lua execution:** Lua scripts are behavior-hook clients, not an alternate scheduler or storage system. Script load/unload/reload is explicit and main-thread only. Script failures, instruction-budget exhaustion, malformed reflected values, and invalid opaque targets are reported through Lua and behavior diagnostics without throwing through scheduler callbacks or stopping unrelated hooks.
- **Scene physics transform sync:** Scene physics systems register while the scene is `Unloaded` or `Loaded`. `FixedPrePhysics` reads scene transforms into static and kinematic bodies, `FixedPhysics` steps Jolt, and `FixedPostPhysics` writes dynamic body transforms back to root scene actors. Dynamic bodies on parented actors are rejected in Phase 10. Static triangle mesh colliders consume CPU vertices/indices only; renderer mesh handles, authored scene automatic collider generation, navigation rebuilds, scripting, and serialization are not part of the current physics path. Terrain colliders enter scene physics only through explicit `TerrainPhysicsColliderAdapter` calls.
- **Scene procedural gameplay:** The procedural App sample defaults to legacy `ActorController` and `BlockingCollisionSystem`. Its experimental scene-character mode is opt-in and requires callers to explicitly create terrain colliders through `TerrainPhysicsColliderAdapter`, prop blockers through scene physics descriptors, and local nav tiles through the existing navigation build/cache path before scene character movement can use them. Scene character movement must report unsupported or missing-tile cases instead of silently falling back to legacy actor commands.
- **Scene render bridge to renderer:** Scene render bridge component handles are transient and must not be serialized. The bridge may create renderer mesh/skinned instances and lights through `SceneRenderBackend`, but renderer handles remain renderer-owned resources. Camera components only build `Renderer::RenderView` data; App still owns bgfx view setup and active-camera selection. Phase 5 uses direct renderer handles only for active resource binding; registry/cache-backed resource resolution is deferred.
- **Authored scene adapter ownership:** The scene authored adapter owns renderer static meshes, renderer materials, and cached texture acquisitions it creates from imported scene CPU data. Callers must release bridge-created renderer instances/lights through `SceneRenderBridge` and release adapter-owned resources through `releaseSceneAuthoredAdapterResources` before shutting down `AssetCache` or Renderer. Imported node hierarchy and local transforms are copied into `Scene`; existing authored scene runtime owners remain unchanged.
- **Animated scene adapter ownership:** The scene animated adapter owns renderer skinned meshes, optional bind-pose meshes, renderer materials, cached texture acquisitions, skeleton records, and animator records it creates from imported animated CPU data. Callers must release bridge-created skinned instances through `SceneRenderBridge` and release adapter-owned resources through `SceneAnimatedModelAdapter::releaseResources` before shutting down `AssetCache` or Renderer. Animator updates run only through explicit adapter calls or an optional `VariableAnimation` scene scheduler system; pre-render renderer sync remains owned by `SceneRenderBridge`. Root-node skinned binding fallback is opt-in sample compatibility behavior, not a default import rule.
- **Asset registry identity:** Asset registry records have two identities: transient `AssetHandle` values for runtime lookup and stable `AssetId` values for durable references. Asset IDs are derived from canonical source path, asset type, and import settings unless an explicit tool/test ID is supplied. Dependency edges store `AssetId`s and resolve to live handles only when the referenced assets are currently registered. Registry metadata is CPU-only; it must not become renderer resource ownership or an implicit import pipeline.
- **Asset registry to cache:** `AssetRegistry` tracks source identity and dependencies; `AssetCache` continues to own renderer mesh/texture reuse and explicit release before renderer shutdown. Registry records may describe assets that are missing, stale, generated, or not loaded into renderer memory.
- **World to renderer:** World objects may hold renderer instance handles, but renderer resources do not own world object lifetime. `World::syncRenderState()` is the transform handoff.
- **World to save:** Save files use `ObjectId`, never `WorldObjectHandle`. Procedural IDs are stable only if chunk coordinate derivation, archetype IDs, and local slot numbering remain stable.
- **Chunk to world:** Chunk streamer owns loaded membership lists. `World` owns the object records. Unloading a chunk must remove its objects from spatial registry and destroy renderer instances through world/chunk cleanup. Budgeted unloads should split navigation removal, object destruction, terrain destruction, render-group destruction, and final dirty marking into separate cooperative work items.
- **Async chunk generation:** Worker jobs may generate `GeneratedTerrainTileData`, `GeneratedChunkProp` descriptors, navigation build geometry, and Detour tile bytes from immutable snapshots. Main-thread commit must create render groups, terrain renderer handles, world objects, renderer instances, spatial entries, and live navigation tiles.
- **Budgeted main-thread work:** Deferrable main-thread work should be represented as cooperative `MainThreadWorkItem`s with a category and priority. Atomic work items may overrun the budget, but systems should split large operations into phases before enqueueing. Critical work may run over budget only when explicitly marked.
- **System inboxes:** Cross-system message flow should use receiver-owned typed inboxes and explicit publish-only sink references when direct calls are too tightly coupled. App or an explicit scheduler owns flush order. Avoid global dynamic event bus registries unless a future feature documents concrete ownership, lifetime, type, threading, and shutdown rules.
- **Terrain to gameplay:** Gameplay height, picking, prop placement, diagnostics, and actor grounding use CPU terrain data, not renderer LOD meshes. Biome terrain-shape fields are save/cache-affecting generation inputs.
- **Terrain dataset identity:** `TerrainSourceHandle` and `TerrainChunkHandle` are transient runtime handles and must not be serialized. Durable terrain identity uses source asset IDs plus source-space chunk coordinates; future chunked serialization and generated caches should use that durable identity, not runtime handles.
- **Terrain chunk serialization prep:** Future serialized terrain chunk files use `TerrainChunkStableIdentity`: source asset ID, source-space chunk coordinate, source type, import settings key, source revision, material revision, chunk resolution, and chunk size. `TerrainSerializedChunkFileMetadata` records schema/payload versions, payload role, identity hash, payload file name, and payload boundary flags. Payload boundaries may describe authoritative heights, edited deltas, material overrides, or derived data references, but they must never contain live runtime handles. T9 defines metadata and validation only; no chunk file reader/writer exists yet.
- **Terrain derived cache:** Terrain derived cache files are disposable generated data. Manifest identity must include durable source/chunk identity, source content or procedural/settings hash, import settings, output kind, output resolution/LOD, and payload versions. Cache helpers may read/write files on workers using immutable manifest and payload snapshots, but cache hits only return CPU payloads; callers still explicitly commit live renderer, navigation, or physics resources on the appropriate thread. Terrain chunk heights, renderer-independent LOD meshes, and renderer-independent terrain physics collider meshes are cache payloads; they must never contain renderer handles, `ScenePhysicsWorld` bodies/colliders, Jolt identifiers, nav handles, App state, or serialized runtime handles.
- **Terrain to renderer:** TerrainSystem owns CPU tiles and creates/destroys renderer terrain handles. Renderer only owns the active draw buffers it is handed. Renderer LOD rebuilds are deferrable visual work: workers generate `TerrainRenderMeshData` through `TerrainRenderLodAdapter` from immutable CPU tile/chunk snapshots, optionally using read-only derived cache hits, but bgfx handle replacement and material/render metadata restoration remain main-thread work. Gameplay height queries continue to use CPU terrain while out-of-budget or pending tiles temporarily keep their previous renderer LOD.
- **CPU-only terrain fixtures:** `TerrainSystem` may be configured not to create renderer resources and may create tiles from explicit height arrays. This mode is for headless tests/tools and must preserve the same CPU height sampling and navigation build data contracts as runtime terrain.
- **Navigation local layer:** `NavigationSystem` only knows loaded nav tiles. Query failure states such as `NoTile`, `NoNearestPoly`, and `NoPath` are normal gameplay outcomes. Missing runtime tiles should become pending cache/worker work, not synchronous Recast builds.
- **Navigation runtime facade:** `SceneNavigationService` reads from existing live navigation systems and may read scene actor transforms through public `Scene` APIs. It does not own or mutate scene hierarchy, chunk streaming, nav tile generation, connectivity, graph, or cache state. Runtime query diagnostics and debug requests are plain Engine data and must not affect pathfinding results.
- **Navigation scene geometry:** Scene navigation source handles are transient and must not be serialized. Source geometry is CPU data in local source space; actor-owned sources use `Scene::worldMatrix` only while snapshotting or refreshing dirty bounds. Authored scene adapter navigation registration is opt-in and uses imported CPU mesh primitives before renderer static mesh handles are created. Dirty chunks are reports for callers to schedule cache/worker rebuilds; the registry must not mutate `NavigationSystem`, enqueue workers, inspect renderer resources, or expose Recast/Detour types.
- **Navigation connectivity:** Connectivity is derived metadata from loaded local nav tiles. Each connected portal stores the exact paired neighbor portal position; debug links and graph routes must use that explicit pair rather than searching for any connected neighbor portal. Connectivity must update after nav tile insert/destroy/clear and must not load or unload chunks. Runtime dirty connectivity updates should use `beginRebuild`/`stepRebuild` so Detour portal sampling is cooperative and budgetable by edge/sample count. The world graph must wait until the active connectivity rebuild handle completes before consuming refreshed connectivity data.
- **World graph:** Coarse graph can generate nodes beyond loaded chunks, but it must not build local nav tiles, stream chunks, or move actors. Runtime graph construction should use immutable terrain/biome/connectivity snapshots on worker threads, then swap plain `WorldNavigationGraphCacheData` into the live graph on the main thread. Loaded neighboring chunks may have multiple graph edges, one per connected portal pair, so route planning can choose a nearby usable portal instead of a single chunk-pair default edge. Large graph rebuilds should be coalesced and recentered only after the camera/route center moves a configurable chunk threshold, not every single chunk crossing.
- **Dirty derived data:** Chunk streaming, terrain LOD rebuilds, explicit nav rebuilds, save/load, and persistence edits are responsible for marking dependent dirty flags. Dirty processing should enqueue coalesced budgeted work instead of immediately rebuilding when the result can wait. Standing still should not rebuild nav tiles, connectivity, graph data, or renderer visibility metadata. If a dirty pass can touch many chunks, it should expose a per-frame cap and report deferred work through debug stats.
- **Navigation cache:** Cache files are derived baseline data, not save data. Manifest identity must include settings and source config inputs that affect generated navigation. Chunks with save-backed object overrides bypass baseline cache unless the user explicitly refreshes cache records through debug tools. Cache file reads/writes should run through worker-safe helpers using immutable settings/manifest snapshots; the live `NavigationCache` facade remains main-thread stats/debug state only. Async chunk jobs may use baseline cache hits to avoid Recast generation, but live tile insertion remains main-thread only. Cache write-through defaults off so disk I/O does not create surprise frame hitches. Terrain nav cache manifests include the border-aware terrain navigation adapter version, border padding, and border sample count so older non-apron seam tiles are stale after the seamless nav generator lands.
- **Terrain to navigation:** Terrain navigation build data is generated through `TerrainNavigationAdapter` from authoritative CPU height chunks at navigation resolution. The adapter can build the Detour tile from a border-expanded sample region while preserving the original durable chunk coordinate and output tile bounds; `NavigationTerrainBuildData::rasterizationBounds` carries the expanded Recast rasterization bounds, while `bounds` remains the runtime tile bounds. The adapter outputs walkable terrain geometry only; App, scene geometry, or other callers append blocker geometry before building Detour tile bytes. Navigation cache identity includes terrain source/import identity, the terrain navigation adapter version, border-apron settings, and scene geometry filter settings so heightmap/procedural/seam-generation changes cannot reuse stale tile bytes.
- **Terrain to physics:** Terrain physics collider payloads are generated through `TerrainPhysicsColliderAdapter` from authoritative CPU height chunks at collider resolution. The adapter creates one static scene physics body/collider per terrain chunk only through explicit caller-owned calls, tracks dirty collider chunks separately from render and navigation outputs, and never derives gameplay collision from renderer LOD meshes.
- **Terrain to scene runtime:** Terrain data reaches scene navigation, scene physics, and future scene serialization only through explicit adapter handoffs. Creating or loading `TerrainDataset` chunks must not implicitly create nav tiles, scene physics bodies/colliders, renderer resources, or serialized chunk files.
- **Terrain material metadata:** Terrain material sets are CPU metadata containing stable layer/rule IDs, PBR texture asset references, scalar factors, tiling/projection hints, and deterministic height/slope/world-position rules. Texture references are tracked as `AssetRegistry` dependencies only. CPU material coverage diagnostics use authoritative terrain heights and CPU-derived slope, never render LOD normals or renderer material handles.
- **Terrain material rendering:** Renderer terrain material-set handles are live renderer resources and must not be serialized. `TerrainMaterialRenderAdapter` may acquire layer textures through `AssetCache` and create a renderer terrain material set, but callers own release ordering before `AssetCache` and renderer shutdown. Terrain shader blending is visual-only; gameplay material coverage remains the CPU T7 rule evaluation contract.
- **Navigation diagnostics:** Tile, portal, and actor command diagnostics are read-only Engine data for debug/tuning. They may explain existing decisions but must not change pathfinding, movement, chunk streaming, persistence, or cache validity by themselves.
- **Actors and navigation:** Actors own local Detour path state and hierarchical route state. Move commands try a direct local Detour path first. If direct local pathing fails, actors request a coarse route; same-chunk failures may use an adjacent-chunk detour that leaves and re-enters through portal pairs. Chunk transitions use paired source/neighbor portal waypoints tagged with their intended chunk, tile-specific waypoint snapping, and short validated seam bridge movement so actors can cross between adjacent loaded tiles without requiring perfect Detour edge stitching. If the required local nav tile is unavailable, the actor waits or fails without teleporting or forcing streaming.
- **Streaming and routes:** Chunk streaming remains camera/App-owned. Hierarchical movement may observe loaded tile availability, but actors must not directly load chunks or build nav tiles.
- **Collision:** Procedural runtime collision uses world CPU bounds and collision flags through `BlockingCollisionSystem`. Scene physics uses `ScenePhysicsWorld` bodies/colliders. Renderer bounds are for rendering/culling and must not become gameplay authority.
- **Reflected writes:** Reflection is an access boundary, not a storage owner. Setters must validate handle kind/owner/generation, value type, finite numeric values, and subsystem invariants before forwarding to public APIs such as `Scene::setLocalTransform`, scene render bridge descriptor setters, `ScenePhysicsWorld` mutators, or `SceneCharacterMovementSystem` mutators. Unsupported or read-only properties fail with typed status rather than silently mutating internal state.
- **Input:** Engine behavior consumes semantic input actions from `EventQueue`. New gameplay controls should be added to `InputMapping`/YAML rather than hardcoded SDL checks. If input starts feeding durable commands into a system-owned inbox, the mapping remains the producer and the receiving system owns the inbox/flush contract.
- **Interactions:** Interaction systems publish/resolve outcomes; App applies world, renderer, and persistence mutations through Engine services.
- **Debug UI:** Dear ImGui owns presentation of debug display/knobs. App supplies modern runtime snapshots, Renderer/DebugUi presents them, and debug knobs must mutate runtime state through public Engine/Renderer APIs rather than legacy subsystem storage or hidden simulation state.
- **Debug draw:** Debug primitives are transient diagnostic geometry. Expensive categories must be capped and report generated/submitted/clipped counts; clipped debug visualization is acceptable and must not affect gameplay, navigation, rendering culling, or persistence.
- **Debug visualization requests:** `DebugVisualizationCollector` output is frame-transient and must not be stored in scene binary files, terrain files, save data, derived caches, or asset manifests. Debug settings may be user preferences later, but Phase 16 collector state is not serialized.
- **Long-frame diagnostics:** Debug builds should preserve enough frame timing context to identify rare hitches. The long-frame recorder is diagnostic only and must not change simulation, streaming, renderer, cache, or navigation behavior.
- **Release debug tools:** `MANUAL_ENGINE_ENABLE_DEBUG_TOOLS` is `1` for Debug and `0` for Release. Release builds stub Dear ImGui, skip debug primitive gathering/submission, and do not link the ImGui target.
- **Picking cadence:** Debug builds continuously update hover picking for inspection. Release builds run picking only for commands that require a fresh target, such as select/interact/place/move clicks.
- **Renderer visibility:** Render layers, visibility flags, render groups, materials, and max draw distance affect submission only. They must not drive simulation, chunk lifetime, or persistence. App-owned visibility metadata updates should be incremental by chunk when possible; full loaded-set reapplies are reserved for global debug setting changes and must be budgeted.
- **Asset cache:** Cache-owned renderer assets must be explicitly released before renderer shutdown. Terrain tile resources are transient and stay outside the cache. `AssetId` and `AssetHandle` are not renderer handles and must not be released through renderer or cache APIs.

## Update Rule

Before adding or changing a feature that touches more than one subsystem, consult this guide and `docs/engine_overview.md`. Update this guide in the same change when work adds:

- a new gameplay system or renderer feature;
- a new public API/interface contract;
- a new initialization/shutdown dependency;
- a new save-facing identity or serialization rule;
- a changed ownership boundary between App, Engine, Renderer, or Assets;
- a new debug control that mutates runtime behavior.
